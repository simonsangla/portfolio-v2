## Problem

Design tokens usually live in a Figma file nobody can consume as code, or in a brittle design-system repo nobody can edit without a pull request. Either way, themeability is slow: a handoff meeting, a copy-paste pass, a round of "can you also export it as Tailwind?"

## Approach

theme-forge is a static, browser-only React/TypeScript app. Seven domains, specified as Cavekit requirements before any code:

- **Product Boundary** — locks the product surface (no backend, no routing, no auth, no CMS, no marketplace).
- **Schema** — canonical theme data model with a variant pair shape (light/dark) and Zod validation surface.
- **Editor** — interactive token controls, built-in presets, undo/redo history, Simple/Advanced color mode.
- **Preview** — live canvas rendering representative UI under the active theme with a light/dark toggle.
- **Export** — pure serialization to JSON, CSS, TS, Tailwind, SCSS, Style Dictionary; copy and download.
- **Persistence** — local-storage save/restore; import from JSON string or file.
- **Widgets** — bounded manifest of 11 themed preview cards; selection persists alongside the active theme.

Stack: Vite 8 + React 19 + strict TypeScript + Zod + Vitest. All state in `localStorage`. Nothing leaves the tab.

## Outcome

Batch B-arch (PR #24, 2026-04-16) locked the Simple/Advanced color mode contract across Editor (R11) and Persistence (R8) with zero source/test/schema changes — architecture-only. Coverage summary: 55 requirements across 7 domains. Widget catalog frozen at 11 IDs. All four validation gates green on merge.
